## Configuration files

| File | Purpose |
| --- | --- |
| `config/config.exs` | Shared compile-time configuration |
| `config/dev.exs` | Development overrides |
| `config/test.exs` | Test overrides |
| `config/prod.exs` | Production overrides |
| `config/runtime.exs` | Runtime configuration from environment variables |

## Compile-time vs runtime

Secrets and environment-dependent values belong in runtime
configuration.

Use `runtime.exs` for:

- Database URL
- Encryption keys
- API secrets
- Deployment-specific credentials

The same release artifact can be reused across environments by changing
environment variables.

## Bootstrap strategy

Initial source and sink definitions should be seeded through
operational workflows, not static configuration files.

Recommended mechanisms:

- A mix task such as `mix claptrap.seed`
- The API itself

<Warning>
  Static config files that duplicate source or sink definitions create
  a split-brain source of truth between configuration files and the
  database.
</Warning>

## Credential storage

Source and sink configurations may include API keys, OAuth tokens, and
webhook secrets. These are stored encrypted at rest using
**Cloak.Ecto**.

## Claptrap.Vault

`Claptrap.Vault` is a GenServer started early in the supervision tree
that manages encryption keys. Ecto schemas transparently encrypt and
decrypt JSONB credential fields on read and write.

## Key management

The encryption key comes from an environment variable
(`CLAPTRAP_VAULT_KEY`).

In production, store that secret in a proper secrets manager:

- AWS Secrets Manager
- HashiCorp Vault
- An equivalent secret distribution mechanism

## Design intent

The architecture draws a clean line:

- Topology and content state live in PostgreSQL
- Secret material is encrypted at rest
- Deployment-specific values are injected at runtime
